# Figma 设计稿落地规范

本文用于约束 Web 前端从 Figma 设计稿还原到代码的默认流程，目标是避免只看局部节点、孤立照搬像素值，导致页面比例、字号、间距在真实应用中失真。

适用范围：`apps/dsa-web/` 中由 Figma 设计稿驱动的页面、布局、组件和交互样式调整。

## 1. 必须先看整页，再看局部

接到 Figma 链接后，先判断链接指向的是整页 frame 还是局部节点。

- 如果链接是整页 frame，先读取整页 frame 的宽高、主要子节点坐标和层级。
- 如果链接是局部节点，先尝试向上查找其所属页面或父级大 frame；如果无法确认整页结构，应要求补充整页节点链接。
- 禁止只读取局部节点后直接照搬其中的字号、padding、卡片高度、阴影和边框。

实施前至少记录以下信息：

- 整页 frame 的宽度和高度，例如 `1440 x 1196`。
- 页面横向结构，例如 `左导航 / 间距 / 历史侧栏 / 间距 / 主内容 / 右边距`。
- 关键区域的 `x / y / width / height`，包括导航、顶部搜索区、侧栏、主内容区和目标组件。
- 目标局部节点相对于整页 frame 的位置和占比。

## 2. 先对齐页面骨架，再实现组件内部

Figma 局部节点里的尺寸只有在页面骨架一致时才能直接使用。实现前必须先判断当前代码中的页面骨架是否和设计稿一致。

需要核对：

- 外层 Shell / Sidebar 宽度是否和设计稿一致。
- 页面主内容的 padding、gap、最大宽度是否和设计稿一致。
- 目标区域的实际渲染宽度是否和 Figma 节点宽度一致。
- 目标组件周围是否有额外容器 padding、缩放、滚动条或边框。

如果页面骨架不一致，优先调整或抽象页面级布局 token，再实现局部组件。不要用缩小字号、减少 padding 等方式掩盖页面级布局偏差。

## 3. Tailwind 与 token 分层

当前 Web 前端大量使用 Tailwind class 表达布局和尺寸。Figma 落地时不要为了追求“像素完全映射”而新建一套平行 CSS 系统；应先判断尺寸属于页面布局、组件密度还是主题语义。

页面骨架 token：

- 左侧导航宽度
- 页面主容器宽度
- 侧栏宽度
- 主内容宽度
- 页面级 gap / padding
- 页面最大宽度和居中策略

页面骨架优先用 Tailwind 的断点、grid/flex、`minmax()`、`max-w-*`、`w-*`、`gap-*`、`px-*` 表达；只有当同一页面骨架会被多处复用，或需要浅色/暗色/断点统一覆盖时，才抽到语义 class 或少量 CSS token。

组件密度 token：

- 卡片高度和内边距
- 标题、正文、辅助文字字号
- 图标尺寸
- 列表行间距
- 状态标签尺寸
- hover / selected 的边框、背景、阴影

组件密度优先复用现有 Tailwind 尺寸、字体和已有语义 class。只有颜色、主题态、复杂状态样式、跨组件复用的密度值才考虑进入 `index.css`。

页面骨架可以随断点变化；组件密度不应随大屏无限放大，只在小屏必要时降级。

## 4. 响应式策略

设计稿宽度是桌面基准，不是所有屏幕的硬编码宽度。

默认策略不是用流体 rem 或 viewport 线性缩放整页，而是用断点切换布局：

- 桌面基准：接近设计稿宽度时，按设计稿的页面骨架比例还原。
- 中等屏幕：允许导航折叠、主内容弹性收缩、侧栏保持固定或进入更紧凑布局。
- 小屏幕：侧栏优先进入抽屉或单栏，不强行保留桌面横向布局。
- 大屏幕：增加外部留白或居中最大容器，不放大组件字号、卡片高度和 padding。

允许使用 `clamp()` 的场景很少，仅限页面级容器宽度、图表高度等确实需要连续变化的元素。正文、按钮、列表卡片、侧栏项默认不要用流体 rem 或 viewport 宽度缩放。

## 5. CSS 变量治理

`apps/dsa-web/src/index.css` 是主题和全局语义样式入口，不是每次 Figma 还原都新增变量的地方。

要求：

- 优先复用已有变量、主题色和语义 class；如果已有变量表达的是同一语义但值不符合设计稿，应修改现有变量值，而不是新增同义变量。
- 少新增变量。只有当样式具备明确语义、会跨组件复用、需要浅色/暗色模式分别覆盖，或 Tailwind class 无法清晰表达时，才新增变量。
- Tailwind 可以直接表达的一次性布局尺寸、间距和断点，不要强行变量化。
- JSX 可以使用 Tailwind 表达结构、布局和局部尺寸；主题颜色、复杂选中态、阴影和跨主题状态优先落到现有 `index.css` 变量或语义类。
- 浅色和暗色模式通过同一变量名覆盖不同值，避免在组件里写主题分支。
- 新增变量前必须先搜索 `index.css` 是否已有同类 token，并在交付说明中说明为什么不能复用。

优先复用或调整这类已有变量：

```css
--home-history-item-bg
--home-history-item-border
--home-history-item-selected-bg
--home-history-item-selected-border
--home-history-item-selected-shadow
--home-history-item-selected-indicator
```

不要为了每个 Figma 节点新增一组一次性变量，例如 `--figma-frame-520-16233-padding`。

## 6. 实施检查清单

改动前：

- 已读取整页 frame，而不是只看局部节点。
- 已记录页面级横向结构和目标区域实际尺寸。
- 已核对当前代码的 Shell、页面 padding、侧栏宽度和主内容宽度。
- 已判断哪些 Figma 尺寸可以直接使用，哪些需要通过布局 token 适配。

改动中：

- 页面骨架优先用 Tailwind 断点、grid/flex 和必要的语义 class 表达。
- 组件内部尺寸优先复用 Tailwind 和已有密度样式；只在跨组件或跨主题时抽 token。
- 颜色、阴影、选中态、hover 态优先复用或调整 `index.css` 现有变量；确需新增时保持少量、语义化。
- 不新增平行 UI 实现，不绕开现有组件体系。

改动后：

- 1440 宽度下整体比例接近设计稿。
- 1280 / 1024 宽度下布局不拥挤、不重叠。
- 移动端无横向溢出，侧栏/列表有合理降级。
- hover、selected、disabled、loading、empty 状态均可检查。
- 暗色模式下背景、边框、文字和阴影仍可读。

## 7. 交付说明要求

交付 Figma 驱动的 Web UI 改动时，说明中必须包含：

- 使用的 Figma 整页节点和局部节点。
- 设计稿页面骨架与当前代码布局是否一致。
- 哪些尺寸按设计稿直接使用，哪些通过响应式策略调整。
- 复用了哪些现有 `index.css` token；如新增 token，说明原因。
- 已验证的屏幕宽度、状态和主题。
